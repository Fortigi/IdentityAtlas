- Added Azure DevOps crawler: sync users, projects, teams, team memberships, security groups, group memberships, Git repositories, and repository-level ACLs from any ADO organization; user access levels (Basic/Stakeholder) are included when authenticating with a PAT
- Azure DevOps wizard walks through PAT credential setup, scope selection, options, and scheduling — the same step model as the Entra ID crawler
- ADO PAT credentials are stored encrypted in the secrets vault using AES-256-GCM envelope encryption — not as plaintext in the config table
- ADO users who originate from Entra ID (e.g. synced via Entra ID AAD-backed accounts) are automatically correlated to existing Entra ID identities using the Identity/IdentityMember model
- Repository ACLs are decoded from ADO security namespace bit masks to human-readable permission labels (Read, Contribute, ManagePermissions, etc.) stored in extendedAttributes
- Resources list now shows the parent resource name (e.g. "in Fortigi-Product") as a subtle subtitle under the resource display name, making identically-named resources in different projects immediately distinguishable
- Fixed a crawler bug where Contains relationships (team→project, group→project, repo→project) were overwriting each other during full sync; they are now sent as a single combined batch
- Fixed group→project relationships missing for some project-scoped groups: the crawler now extracts the project GUID directly from the group's domain field (vstfs:///Classification/TeamProject/&lt;guid&gt;) instead of relying on a name lookup from principalName, which was unreliable for groups with special characters in the project name
- Fixed projects fetch being capped at 200: the ADO projects API returns the continuation token as a response header (x-ms-continuationtoken), not in the body, so Invoke-AdoApi never paginated it; switched to a dedicated loop that reads the header and fetches all pages
- Added best-effort resolution of project names for groups whose parent project is not in the accessible projects list: after syncing groups the crawler now individually calls _apis/projects/&lt;guid&gt; for each unknown project GUID; projects that resolve are synced as AzureDevOpsProject resources so their child groups show the correct parent name; projects that 401/403 are silently skipped
- Fixed the "Member Of" section in resource detail showing a GUID instead of the parent resource name, and missing resource type — caused by PostgreSQL lowercasing unquoted SQL aliases
- Renamed "Member Of" to "Parent Resources" in resource detail — the term is more accurate for a generic parent-child resource hierarchy and avoids Active Directory group-membership connotations; the Relationship column now reads "Contained in" instead of "Contains" when viewed from the child's perspective
- Fixed "Parent Resources (1)" collapsing to "(0)" with no rows when expanded — the count query counted relationship rows directly while the detail query used INNER JOIN, which silently dropped rows for parent resources not yet in the Resources table (e.g. inaccessible ADO projects); switched to LEFT JOIN so all relationships are shown regardless of whether the parent resource record exists
- Added PAT access-scope warning in the Azure DevOps crawler wizard: the credentials step now explains that a PAT inherits the permissions of its creator, so projects the user cannot access will produce groups without resolvable parent names, and advises granting Project Reader on all projects for full coverage
- Resources list now shows "parent project not accessible" (with hover tooltip) instead of a bare GUID for groups whose parent project could not be resolved due to credential scope; the tooltip explains how to fix it
- Parent Resources section in resource detail now shows the raw project GUID in italic with a hover tooltip instead of a clickable link when the parent resource could not be resolved
- Azure DevOps crawler now emits a clear access-scope warning at the end of each run listing how many projects were referenced by security groups but could not be resolved, with guidance on how to grant the necessary permissions
- Resources list now includes a "Source System" filter in the Filters bar, allowing resources to be filtered by which crawler they originate from
- Fixed Filters dropdown appearing white with near-invisible text in dark mode — select elements inside the filter bar now use explicit dark backgrounds instead of transparent
