# Identity Atlas v4.0 (formerly FortigiGraph)

**Branch:** `feature/universal-resource-model`
**Date:** 2026-03-31
**Base:** `dev`

## Summary

Major data model refactoring to support multi-system authorization data. The previous model was Entra-ID-centric (GraphGroups, GraphUsers). The new model introduces a universal hierarchy: **Systems > Resources/Principals > OrgUnits > Identities** — enabling import of permissions from any authorization source (Entra ID, SharePoint, Azure RBAC, SAP/Pathlock, DevOps, file shares, etc.).

## Risk Scoring Enhancements

### Resource-Type-Aware Scoring
- **Type multipliers**: EntraDirectoryRole (1.5x), EntraAppRole (1.2x), EntraGroup (1.0x baseline)
- **Type-specific structural signals**: Directory role patterns (Global Admin +25), app role patterns (.ReadWrite +10)
- **Type-specific propagation**: DirectoryRole 40%, AppRole 35%, Group 30%
- **LLM-determined multipliers**: `New-FGRiskProfile` generates org-specific multipliers via LLM

### Scoring Priority Chain
1. Config file override (explicit user settings)
2. Risk profile from SQL (LLM-determined)
3. Hardcoded defaults

## Design Patterns

### Core + JSON
Resources and Principals use the same pattern: frequently-queried attributes as real SQL columns, system-specific attributes in `extendedAttributes` JSON. This keeps the schema clean while allowing flexible extension.

### Backward Compatibility
All queries use try/catch fallback: prefer new tables (Resources, Principals), fall back to old tables (GraphGroups, GraphUsers). Both old and new tables coexist during migration.

### Deterministic GUIDs
OrgUnits and AppRole resources use MD5-based deterministic GUIDs for idempotent syncs.

## Changes in this branch

- **Risk scoring: full port of the v4 4-layer engine to postgres.** The lightweight v5 scorer was only running Layer 1 (direct classifier match) + a 5-point small-group bonus, which missed almost everything interesting. [app/api/src/riskscoring/engine.js](app/api/src/riskscoring/engine.js) now implements all four layers with v4's exact weights (0.50 / 0.20 / 0.10 / 0.20) and layer caps (membership 40, structural 25):
  - **Layer 2 — Membership analysis** (cap 40): small-group bonus, no-owner penalty, broad-access-footprint (in >15 groups → +3 per 3 above), member-of-high-risk-group (direct ≥70 → +15), many-ownerships (>3 → +5), **span of control** (≥5 direct reports → +3-15), **large org subtree** (BFS through `Principals.managerId` — ≥10 total reports → +5-15), **manager of high-risk direct reports** (+5 per, cap 15).
  - **Layer 3 — Structural hygiene** (cap 25): no description (+3), mail-enabled security group (+3), role-assignable group (+15), dynamic membership rule on (+3), disabled account with active memberships (+5), guest user (+5), stale sign-in >90 days (+10).
  - **Layer 4 — Cross-entity propagation**: one-pass, v4 dampening factors — user inherits 30% of max group pre-prop score, group inherits 25% of max member pre-prop score. No recursion.
  - **Non-production discount**: groups matching `_(ACC|TST|DEV|ONT|STG|SBX|UAT|QA)`, single-letter OTAP markers `_[ATDO]_` / `_[ATDO]$`, or keywords `acceptat|develop|ontwikkel|staging|sandbox|non-prod|pre-prod` get their direct score ×0.25 (floor 5). Mirrors v4 so dev/test groups don't dominate the Medium+ list.
  - Hierarchy signals gracefully degrade to 0 when `Principals.managerId` is not populated (via an `hierarchyAvailable` flag computed up front).
  - Full per-layer explanations are persisted into `RiskScores.riskExplanation` — the detail page's "Show Details" expansion now shows exactly which signals fired and how much each contributed.
- **Entra crawler: pulls manager, signInActivity, userType, externalUserState.** [Start-EntraIDCrawler.ps1](tools/crawlers/entra-id/Start-EntraIDCrawler.ps1) `/users` `$select` list expanded to include `userType`, `signInActivity`, `externalUserState`, plus `$expand=manager($select=id)` in one round-trip. These feed the scoring engine's guest/stale/hierarchy signals. Persists into `Principals.managerId` (column-generic ingest) and `Principals.extendedAttributes.{userType, signInActivity, externalUserState}` (jsonb). After a re-crawl on the HBR tenant: 4,069/4,464 users have managerId, 4,065/4,464 have signInActivity, 4,464/4,464 have userType.
- **Scoring engine: match-exclusion guardrail.** [engine.js isExcludedMatch()](app/api/src/riskscoring/engine.js) drops any classifier match whose matched-text hits `nieuws/news/newsletter/mededeling`, `vergaderruimte/meeting room/leslokaal/classroom`, `calendar editor/kalender/agenda beheer/room reservation`, `shared mailbox/postbus`, `distribution list/distributielijst`, etc. The LLM prompt asks the generator to avoid these but doesn't always obey. This is a hard-coded safety net that runs after regex match — e.g. `17.128 Harbour Master room (leslokaal) MO` no longer matches the Harbour Master classifier on its description.
- **Resource clustering: restored and wired into every scoring run.** Ported v4's `Save-FGResourceClusters` as `buildClusters()` in [engine.js](app/api/src/riskscoring/engine.js). Two-pass: classifier-based (resources matching the same classifier form one cluster) and name-stem fallback (normalised stem like "domain-admins" groups `SG_DomainAdmins_P` and `GRP-Domain-Admins-TST`). Aggregate score = `0.6 × max + 0.4 × avg`. Wipes and rewrites `GraphResourceClusters` + `GraphResourceClusterMembers` in a transaction. On HBR: 267 clusters built (8 classifier + 259 stem), top classifier clusters are "Vessel Traffic Services and Maritime Traffic Control" (23 members, agg 52 Medium), "ISPS Port Security and Facility Security Officers" (4, 50 Medium), "Domain and Infrastructure Administrators" (1, 48 Medium), "Crisis Management and Emergency Response Teams" (5, 47 Medium).
